## 1. Functional Test Cases (Happy Path)

These test cases verify that the feature works as expected under normal conditions.

* **Successful Upload:** Verify that a user can successfully upload a valid file format (e.g., `.pdf`, `.jpg`, `.png`) within the allowed size limit.
* **File Selection via Browse:** Verify that clicking the "Browse" or "Upload" button opens the local file explorer and allows file selection.
* **Drag and Drop:** Verify that dropping a valid file directly into the designated drop zone successfully uploads it.
* **File Name Display:** Verify that the correct file name and extension are displayed on the UI after selection but before hitting "Submit".
* **Progress Bar:** Verify that a progress bar or loading spinner appears during the upload process and reaches 100% upon completion.
* **Success Message:** Verify that a clear success notification (e.g., "File uploaded successfully!") is displayed post-upload.

---

## 2. Boundary & Validation Test Cases (Negative Path)

These test cases ensure the system gracefully handles invalid inputs and enforces restrictions.

### File Size Restrictions

* **Maximum Limit:** Attempt to upload a file that is exactly at the maximum allowed limit (e.g., 5MB) $\rightarrow$ *Should succeed.*
* **Exceeding Maximum Limit:** Attempt to upload a file that exceeds the maximum limit (e.g., 5.1MB) $\rightarrow$ *Should fail with an appropriate error message.*
* **Zero-Byte File:** Attempt to upload an empty/0-KB file $\rightarrow$ *Should fail or show a validation error.*

### File Type & Extension Restrictions

* **Unsupported Formats:** Attempt to upload an invalid file extension (e.g., uploading an `.exe` or `.mp3` when only images are allowed) $\rightarrow$ *Should be blocked with an error message.*
* **MIME Type Spoofing:** Change the extension of a malicious executable from `.exe` to `.png` and attempt to upload $\rightarrow$ *The system backend should detect the true MIME type and reject it.*
* **No Extension:** Attempt to upload a file that has no extension (e.g., just `filename`) $\rightarrow$ *Should be rejected if it doesn't match allowed types.*

---

## 3. UI, UX, & Edge Cases

These test cases check the user experience and unusual user behaviors.

* **Special Characters in Filename:** Upload a file with spaces or special characters in its name (e.g., `my_photo!@#$.png`) $\rightarrow$ *Should either upload successfully or safely sanitize the filename.*
* **Duplicate File Upload:** Upload a file with the exact same name as a previously uploaded file $\rightarrow$ *The system should either overwrite it, append a number (e.g., `file(1).png`), or prompt the user.*
* **Cancel Upload:** Verify that the user can cancel an in-progress file upload, and that the system halts the transfer immediately.
* **Multiple File Uploads:** If the system supports multi-upload, select and upload multiple files simultaneously $\rightarrow$ *Verify all upload correctly or fail gracefully if the maximum count is exceeded.*
* **File Preview:** For image uploads, verify that a thumbnail preview of the image is rendered correctly after upload.

---

## 4. Security & Performance Test Cases

Critical checks to ensure the application remains secure and responsive.

| Test Case Objective | Expected Behavior |
| --- | --- |
| **Malware/Virus Scanning** | Upload a file containing a known virus or malware string. The system's antivirus layer should block and quarantine it. |
| **Large File Performance** | Monitor system memory and CPU utilization while uploading a file at the maximum allowed size limit to ensure no memory leaks occur. |
| **Network Interruption** | Disconnect the internet connection midway through a large file upload. The system should pause gracefully, show a network error, or resume if supported. |
| **XSS Injection via Filename** | Name a file `<script>alert('XSS')</script>.jpg` and upload it. The UI must render the text safely without executing the JavaScript. |
