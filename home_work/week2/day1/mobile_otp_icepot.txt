Testing an OTP (One-Time Password) login flow on a mobile application or website requires a structured approach to ensure both security and a smooth user experience.

Here is a comprehensive checklist of the critical test cases you should run, broken down by scenario:

---

### 1. Happy Path (The Ideal Flow)

* **Successful Delivery:** Verify the OTP arrives via the requested channel (SMS, Email, or WhatsApp) within the expected timeframe (usually under 30 seconds).
* **Correct Redirection:** Ensure that entering the valid OTP successfully logs the user in and routes them to the correct landing page or dashboard.
* **Auto-Read/Auto-Fill:** If the mobile app supports it, verify that the OS prompts the user to auto-fill the OTP directly from the incoming SMS.

### 2. Validation & Edge Cases

* **Incorrect OTP:** Enter an invalid or typo-ridden OTP. Verify that a clear error message appears and the user remains on the verification screen.
* **Expired OTP:** Wait for the OTP expiration time to lapse (e.g., 2 or 5 minutes) and attempt to submit it. It should be rejected as expired.
* **Alphanumeric vs. Numeric:** If your OTP system only uses numbers, try typing letters or special characters to ensure the input field restricts them or throws a validation error.
* **Incomplete OTP:** Try submitting only 3 digits of a 4-digit or 6-digit OTP to ensure the "Submit" button is either disabled or triggers an "Incomplete code" error.

### 3. Resend & Rate Limiting (Security)

* **Resend Timer:** Verify that a visual countdown timer prevents the user from clicking "Resend OTP" immediately. The button should only become active after the timer hits zero.
* **Brute-Force Protection:** Attempt to enter incorrect OTPs multiple times in a row. The system should lock the user out or require a completely new OTP after a set number of failed attempts (usually 3 to 5).
* **Request Throttling:** Rapidly request multiple OTPs to verify that the backend limits the number of SMS messages a single phone number or IP address can request within an hour to prevent high SMS gateway costs and spamming.

### 4. Technical & Network Interruption

* **Network Switching:** Request an OTP on Wi-Fi, then quickly turn off Wi-Fi to force the mobile device onto cellular data before entering the code. The session should not break.
* **App In the Background:** Minimize the app to check your text messages for the code, then reopen the app. Ensure the app didn't restart or wipe out the phone number you previously entered.
* **Interrupted State:** Lock the device or take a phone call while on the OTP screen, then unlock/return to the app to ensure the state is preserved.

---

> **Testing Tip:** When testing in a staging or QA environment, work with your backend developers to set up a **"White-listed / Mock OTP"** (e.g., entering `123456` always works for specific test phone numbers). This saves money on SMS gateway charges and keeps your automated test suites running smoothly without relying on physical SIM cards.
